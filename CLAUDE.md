# 코딩 규칙 (aq2_qrisproject)

이 프로젝트의 코드는 아래 규칙을 따른다. 신규 작성·수정 시 항상 준수하며, 기존 코드도
손대는 김에 규칙에 맞춰 정리한다. 규칙은 인간이 읽고 유지보수하기 좋은 코드를 최우선 목표로 한다.

## 1. OOP · 모듈화

- **책임 하나 = 클래스 하나 = 파일 하나**. 계층별로 디렉터리를 나눈다:
  `data/`(로딩) · `indicator/`(지표) · `strategy/`(전략) · `backtest/`(체결·성과) · `report/`(리포트).
- 각 계층은 **추상 기반 클래스**(`abc.ABC` + `@abstractmethod`)로 인터페이스를 고정하고,
  구현체는 그 인터페이스만 맞춘다. 예: `Indicator.compute()`, `Strategy.generate_signals()`.
- **관심사 분리**를 지킨다. 전략은 신호만 만들고, 체결·비용·성과는 `backtest` 계층이 맡는다.
  한 클래스가 다른 계층의 세부를 알게 되면 잘못 나눈 것이다.
- **의존성 주입**을 기본으로 한다. 협력 객체(로더·지표·엔진)는 생성자 인자로 받아 교체·테스트가
  쉽게 한다. `Pipeline` 처럼 조립은 한곳(오케스트레이터)에 모은다.
- **파라미터는 생성자에서**, 조정값은 `config.py` 의 `@dataclass` 로 모은다.
  코드 수정 없이 `config.json` 값만 바꿔 재실행되도록 유지한다.
- 데이터 묶음은 `@dataclass`(예: `Signals`)로, 가변 전역 상태는 두지 않는다.

## 2. 로깅 (레벨 체계)

- **`print()` 금지, 표준 `logging` 모듈 사용.** 모듈 상단에서
  `logger = logging.getLogger(__name__)` 로 로거를 얻고 이 로거로만 출력한다.
- 로그 초기화(핸들러·포맷·레벨)는 **진입점 한곳**(`main.py`)에서만 한다. 라이브러리성 모듈은
  설정하지 않고 로거만 사용한다(중복 핸들러 방지).
- **레벨을 의미에 맞게** 사용한다:
  - `DEBUG`  — 개발·디버깅용 상세(중간 계산값, 봉 수 등). 평상시엔 안 보임.
  - `INFO`   — 정상 진행 마일스톤(로드 완료, 전략 목록, 종목별 결과, 리포트 경로).
  - `WARNING`— 진행은 되나 주의가 필요(가중 합 ≠ 1.0, 워밍업 부족, 종목 스킵).
  - `ERROR`  — 해당 작업 실패(종목 로드 실패 등). 예외는 `logger.exception()` 으로 트레이스 포함.
  - `CRITICAL`— 파이프라인 전체 중단.
- **포맷은 사람이 읽기 쉽게**: `시각 · 레벨 · 모듈 · 메시지`.
  예: `logging.Formatter("%(asctime)s [%(levelname)-7s] %(name)s: %(message)s", "%H:%M:%S")`
- 표(종목별 결과·유니버스 평균)처럼 **정렬이 중요한 출력은 폭 지정 포맷**을 유지한다
  (`f"{code:<5} {name:<26} {val:>7.1f}%"`).
- 메시지 앞의 `[config]`·`[data]` 같은 수동 태그는 로거 이름(`%(name)s`)으로 대체한다.

## 3. 가독성

- **한글 docstring·주석**을 유지한다(모듈·클래스·공개 메서드에 docstring 필수).
- docstring 은 **무엇을 왜** 하는지 먼저 쓰고, `Args:` / `Returns:` / `Attributes:` 로 계약을 명시한다.
- 섹션 구분선으로 파일 내부 구조를 드러낸다: `# ── 조립 ─────────────`.
- 함수는 한 화면 안에 들어오게 짧게. 깊은 중첩보다 이른 반환(early return)을 쓴다.
- 이름은 의도가 드러나게(`target_long`, `warmup_bars`). 축약·매직넘버는 상수나 config 로 뺀다.

## 4. 주석

- **왜(의도·근거)를 설명**한다. 코드가 그대로 말하는 "무엇"은 반복하지 않는다.
  예: `# EMA 스무딩(whipsaw↓)`, `# i+1 시가 체결 → 룩어헤드 방지`.
- 비직관적 결정, 도메인 규칙(히스테리시스·워밍업·룩어헤드), 트레이드오프에 주석을 남긴다.
- 주석과 코드가 어긋나지 않게, 코드를 고치면 주석·docstring 도 함께 고친다.

## 5. 기타

- 파일 상단에 `from __future__ import annotations` 를 두고 타입 힌트를 붙인다.
- 실행: `python main.py` (설정은 `config.json`, 없으면 내장 기본값 폴백).

**제안서 통합 문단(넓고 얕은 정찰 — 하나의 경사 규칙).** 문턱 52 + 부분 충전은 저확신 자산을

- 신규 방어주가 50~60 대: 경기소비재 59.5 · 운송 58.9 · 미국유틸리티 57.7 · 미국에너지 57.1
- 문턱 60 이면 방어주가 자격 미달 → 주도주 잔류. 문턱 50 이면 방어주가 랭킹에서 주도주를 축출.

**MDD 를 통제하는 것은 오직 이 문턱이다**: 문턱 60 → −12.7 / 문턱 50 + 보유 만충 → −17.6 /
문턱 60 + 보유 축소 → −12.7(축소는 MDD 를 개선하지 못함) / 문턱 50 + 보유 축소 → −12.7(상쇄일 뿐).
→ 동결값이 문턱을 8점만 내린 이유. 원설계(이진 60/45)가 옳았음을 확인·확장한 결과다.

**슬롯은 충전율과 무관하게 소비된다** — 20%만 채우는 종목도 슬롯 하나를 차지해 기득권자를 밀어낸다.
그래서 하한충전율을 아주 낮게 잡으면 '노출은 없는데 자리는 뺏는' 최악 조합이 된다.

## 기록된 실수 (숨기지 않는다 — 검증 루프가 작동한 사례로 발표 자산)

1. **규율 5 위반**: 경사를 진입·유지에 동시 적용해 두 메커니즘을 한 번에 바꿨다. 분리 측정 전까지
   결과 해석이 불가능했다.
2. **오귀인**: "보유 축소가 MDD 를 지킨다"고 결론냈으나, 분리해보니 문턱 인하 피해를 상쇄한 것뿐이었다.
   MDD 통제는 문턱 단독의 몫.
3. **전달 체인 오염**: 채팅→코드 전달문에서 이전 설계 제안 문구("진입 문턱 60 유지")가 측정 결과의
   서술 위에 잔존해 동결 구성을 잘못 기술. 코드 쪽 실증 대조(문턱 60 구성은 전체 Calmar 0.90 으로
   기각 대상)로 고착 전 차단됨.
4. **검증 없는 근거 서술**: 411060 parquet 커밋의 사유를 "클론 즉시 재현"으로 적었으나, 실제로
   클론해 보니 나머지 39종이 전부 gitignore 라 빈 클론은 이 파일이 있어도 못 돈다. 실제 효용은
   "이미 캐시를 가진 팀원이 pull 했을 때 411060 하나 때문에 막히지 않는 것"뿐. 커밋 메시지 amend.
   → **3번과 같은 종류다: 검증 없이 쓴 근거는 남의 것이든 내 것이든 똑같이 틀리고, 두 번 다 코드
   실증이 잡았다.** 실험 규율이 작동한 사례로 발표 Q&A 에 쓴다.

전체 검증 항목·수치·미검증 한계는 `docs/verification_log.md` 참조.
